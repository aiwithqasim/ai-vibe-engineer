## 1. Security

### 🔴 S-1 — HTTP-Referer hardcoded to `localhost`

**File:** `web/app/api/digital-twin/route.ts`, line 139

```ts
"HTTP-Referer": "http://localhost:3000",
```

This header tells OpenRouter which site is making requests. In production it will always report `localhost`, which will cause the request to fail or be misattributed and may violate OpenRouter's terms.

**Remediation:** Replace with a `NEXT_PUBLIC_SITE_URL` environment variable, e.g.:

```ts
"HTTP-Referer": process.env.NEXT_PUBLIC_SITE_URL ?? "http://localhost:3000",
```

Set `NEXT_PUBLIC_SITE_URL=https://yourdomain.com` in the production environment.

---

### 🔴 S-2 — Custom filesystem `.env` parser is bypassing Next.js security model

**File:** `web/app/api/digital-twin/route.ts`, lines 53–91

```ts
function readApiKeyFromEnvFile(filePath: string): string | null { ... }
function getOpenRouterApiKey(): string | null { ... }
```

This function reads raw `.env` files from the filesystem using `fs.readFileSync` at **request time** and traverses `../` parent directories. This is problematic because:

- Next.js natively loads `.env` and `.env.local` at **build/start time** — this custom logic is redundant.
- Walking parent directories (`../`) risks reading `.env` files outside the project boundary.
- Doing filesystem I/O inside an API route handler on every request is slow and risky.

**Remediation:** Remove `readApiKeyFromEnvFile` and `getOpenRouterApiKey` entirely. Create a `.env.local` file in the `web/` folder:

```
OPENROUTER_API_KEY=your_key_here
```

Next.js loads this automatically. Then simplify the key retrieval to:

```ts
const apiKey = process.env.OPENROUTER_API_KEY ?? null;
```

---

### 🟠 S-3 — No server-side input length validation

**File:** `web/app/api/digital-twin/route.ts`

The frontend enforces `maxLength={500}` on the input element, but the API route has no equivalent validation. An attacker bypassing the browser UI can send arbitrarily long messages, inflating token usage and cost.

**Remediation:** Add a server-side check before calling OpenRouter:

```ts
const MAX_CONTENT_CHARS = 1000;
const incomingMessages = body.messages
  .filter(...)
  .map((m) => ({ ...m, content: m.content.slice(0, MAX_CONTENT_CHARS) }))
  .slice(-12);
```

---

### 🟠 S-4 — No rate limiting on the API route

**File:** `web/app/api/digital-twin/route.ts`

The `/api/digital-twin` endpoint has no rate limiting. Anyone can send unlimited requests, exhausting your OpenRouter credit.

**Remediation:** Add rate limiting. The simplest option on Vercel is their built-in Edge Rate Limiting or Upstash Redis via `@upstash/ratelimit`. As a minimum, add a `Cache-Control: no-store` header and document the missing rate limit as a deployment prerequisite.

## 2. Typos and Bugs

### 🔴 B-1 — Typo: `OPENROOTER_API_KEY` (double-O)

**File:** `web/app/api/digital-twin/route.ts`, lines 63, 76

```ts
if (key === "OPENROUTER_API_KEY" || key === "OPENROOTER_API_KEY") {
...
if (process.env.OPENROOTER_API_KEY) return process.env.OPENROOTER_API_KEY;
```

`OPENROOTER` is misspelled with a double-O. This means the fallback key name will never match any correctly-named environment variable. If a user happened to set `OPENROOTER_API_KEY` by mistake, the typo would silently succeed. Otherwise it adds silent dead code.

**Remediation:** Remove all references to `OPENROOTER_API_KEY`. Once S-2 is fixed, this code is deleted entirely.

---

### 🟠 B-2 — Typo: `.emv` instead of `.env`

**File:** `web/app/api/digital-twin/route.ts`, lines 80–81

```ts
path.resolve(process.cwd(), ".emv"),
path.resolve(process.cwd(), "../.emv"),
```

`.emv` is a misspelling of `.env`. These paths will never resolve to anything and are dead candidates silently failing on every cold start.

**Remediation:** Remove once S-2 is resolved. If a manual file check is kept, fix to `.env`.

---

### 🟡 B-3 — `page.module.css` is unused boilerplate

**File:** `web/app/page.module.css`

This file is auto-generated by `create-next-app` and contains styles for the default scaffold page. `page.tsx` does not import it and none of its classes are used. It defines conflicting variable names (`--text-primary`, etc.) that could confuse future developers.

**Remediation:** Delete `web/app/page.module.css`.

## 3. Code Quality

### 🟡 Q-1 — Unnecessary `useMemo` for a trivial boolean

**File:** `web/app/components/DigitalTwinChat.tsx`, lines 24–27

```tsx
const canSend = useMemo(
  () => input.trim().length > 0 && !isLoading,
  [input, isLoading],
);
```

`useMemo` is designed for expensive computations. A simple boolean derivation from two state values does not benefit from memoization and adds cognitive overhead.

**Remediation:** Replace with a plain derived constant:

```tsx
const canSend = input.trim().length > 0 && !isLoading;
```

---

### 🟡 Q-2 — API response type is cast, not validated

**File:** `web/app/components/DigitalTwinChat.tsx`, line 50

```tsx
const data = (await response.json()) as { reply?: string; error?: string };
```

Using `as` silently asserts a shape without verifying it at runtime. If the server returns an unexpected payload, `data.reply` will be `undefined` and the error handling path may miss it.

**Remediation:** Either use a runtime-safe check, or add a lightweight validation helper:

```ts
function isValidReply(data: unknown): data is { reply: string } {
  return typeof data === "object" && data !== null && typeof (data as Record<string, unknown>).reply === "string";
}
```

---

### 🟡 Q-3 — `sanitizeAssistantReply` collapses all newlines into a single line

**File:** `web/app/api/digital-twin/route.ts`, lines 103–125

```ts
const lines = cleaned.split(/\r?\n/).map((line) => line.trim()).filter(Boolean);
cleaned = lines.join(" ");
```

This joins every paragraph and list item onto one continuous line, discarding any intentional paragraph structure in the AI reply. For longer or structured answers this will render as a wall of text.

**Remediation:** Consider preserving paragraph breaks (double newlines) while only collapsing excessive blank lines:

```ts
cleaned = cleaned.replace(/\n{3,}/g, "\n\n").trim();
```

If single-line output is truly the goal, this is fine — but it should be a documented design decision.

`careerJourney` and `futurePortfolioLinks` are defined inline inside the `Home` component. This mixes data and presentation, making updates harder and the file longer than necessary.

```ts
// web/data/profile.ts
export const careerJourney = [ ... ];
export const futurePortfolioLinks = [ ... ];
```

Then import in `page.tsx`. This also makes the data easier to feed into the system prompt later.

### 🟡 Q-5 — System prompt is a module-level constant; profile is duplicated

## 4. Architecture and Design

### 🟠 A-1 — No error boundary around `DigitalTwinChat`

**File:** `web/app/page.tsx`, line 170

```tsx
<DigitalTwinChat />
```

If the chat component throws a runtime error, the entire page will crash. Since this is a client component making async calls, this is a realistic risk.

**Remediation:** Wrap the component in a React error boundary (or Next.js `error.tsx` segment):

```tsx
<ErrorBoundary fallback={<p>Chat is temporarily unavailable.</p>}>
  <DigitalTwinChat />
</ErrorBoundary>
```

---

### 🟡 A-2 — Chat history is not persisted

**File:** `web/app/components/DigitalTwinChat.tsx`

State is purely in-memory. A page refresh wipes the full conversation, which can frustrate users mid-session.

**Remediation:** Persist `messages` to `localStorage` with a `useEffect`:

```ts
useEffect(() => {
  localStorage.setItem("chat-history", JSON.stringify(messages));
}, [messages]);
```

Restore on mount:

```ts
const [messages, setMessages] = useState<ChatMessage[]>(() => {
  try {
    const saved = localStorage.getItem("chat-history");
    return saved ? JSON.parse(saved) : [starterMessage];
  } catch {
    return [starterMessage];
  }
});
```

---

### 🔵 A-3 — `max_tokens: 220` may be too low for detailed answers

**File:** `web/app/api/digital-twin/route.ts`, line 145

At 220 tokens, a thoughtful answer can be cut mid-sentence. This is compounded by the 90-word hard cap in `sanitizeAssistantReply`.

**Remediation:** Raise `max_tokens` to at least `350` and reconsider the 90-word cap. If brevity is the goal, enforce it via the system prompt instruction rather than hard string slicing.

## 5. Accessibility

### 🟡 AC-1 — Screen-reader announcement for loading state is absent

**File:** `web/app/components/DigitalTwinChat.tsx`, lines 86–91

```tsx
{isLoading ? (
  <article className="chat-bubble chat-ai">
    <p className="chat-role">Digital Twin</p>
    <p>Thinking...</p>
  </article>
) : null}
```

The `aria-live="polite"` region on `.chat-shell` will announce the "Thinking..." bubble, but there is no equivalent announcement when loading finishes and the reply appears. Assistive technology users may not know a reply has arrived if they have navigated away.

**Remediation:** Add `aria-atomic="false"` to the chat shell (already the default) and ensure new `<article>` messages are appended rather than replacing existing content, which they are. This is already handled well; the only improvement is to add a visually-hidden status message:

```tsx
<p className="sr-only" aria-live="assertive">
  {isLoading ? "Digital Twin is thinking" : "Reply received"}
</p>
```

---

### 🔵 AC-2 — Send button has no visible focus ring in custom CSS

**File:** `web/app/globals.css`, lines 296–309

The `.chat-form button` styles do not include a `:focus-visible` rule. Browser defaults may supply one, but it will be inconsistent across browsers and may be suppressed by `outline: none` elsewhere.

**Remediation:** Add:

```css
.chat-form button:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}
```

## 6. Performance

### 🔵 P-1 — No `loading.tsx` or `Suspense` for the chat section

The `DigitalTwinChat` component loads inline with the rest of the page. While it is lightweight today, adding `Suspense` prepares the page for future lazy-loading or streaming.

**Remediation:** Add `web/app/loading.tsx` or wrap `<DigitalTwinChat />` in `<Suspense fallback={<p>Loading chat...</p>}>`.
